`xviv` is a command-line controller for Xilinx Vivado and Vitis. It replaces the Vivado GUI workflow for IP creation, block design management, synthesis, and MicroBlaze firmware deployment. All project settings live in a single `project.toml` file.

<!--more-->

## Installation

```bash
pip install -e .
# Enable shell tab-completion (zsh)
echo 'eval "$(register-python-argcomplete xviv)"' >> ~/.zshrc
```

Vivado and Vitis must be installed separately. The default paths are:

```
/opt/Xilinx/Vivado/2024.1
/opt/Xilinx/Vitis/2024.1
```

Override them in `project.toml` if your installation is elsewhere.

## project.toml

Every `xviv` command reads `project.toml` from the current directory (or the path passed with `--config`). A minimal file looks like this:

```toml
[vivado]
path        = "/opt/Xilinx/Vivado/2024.1"
max_threads = 8
mode        = "batch"          # batch | tcl | gui

[vitis]
path = "/opt/Xilinx/Vitis/2024.1"

[build]
dir         = "build"
ip_repo     = "build/ip"
bd_dir      = "build/bd"
wrapper_dir = "build/wrapper"

[sources]
rtl     = ["srcs/rtl/**/*.sv", "srcs/rtl/**/*.v"]
wrapper = ["build/wrapper/**/*.sv"]

[fpga]
part       = "xc7z020clg400-1"
board_part = "tul.com.tw:pynq-z2:part0:1.0"
board_repo = "/opt/board_files"
```

### Multiple FPGA targets

If your project targets more than one board, define named targets:

```toml
[fpga.z2]
part       = "xc7z020clg400-1"
board_part = "tul.com.tw:pynq-z2:part0:1.0"

[fpga.u50]
part = "xcu50-fsvh2104-2-e"
```

Reference a named target from an IP, BD, or synthesis entry with `fpga = "z2"`.

## IP cores

### Defining an IP

Add a `[[ip]]` entry for each custom IP:

```toml
[[ip]]
name    = "ip_systolic"
vendor  = "myorg.com"
library = "hls"
version = "1.0"
top     = "ip_systolic"
rtl     = ["srcs/rtl/systolic/**/*.sv"]
xdc     = ["srcs/xdc/systolic.xdc"]
hooks   = "scripts/ip/ip_systolic_1_0.tcl"
```

| Key              | Purpose |
|---|---|
| `name`           | Identifier used in all `xviv --ip` commands |
| `top`            | HDL top module name (defaults to `<name>_wrapper`) |
| `rtl`            | Glob patterns for source files, relative to `project.toml` |
| `xdc`            | Constraint files for normal synthesis |
| `xdc_ooc`        | Constraint files for out-of-context synthesis |
| `hooks`          | TCL hooks file path (auto-generated by `ip-config`) |
| `create-wrapper` | Set `true` to auto-generate a SystemVerilog wrapper that flattens interface ports |

### Generating the hooks file

```bash
xviv ip-config --ip ip_systolic
```

This creates `scripts/ip/ip_systolic_1_0.tcl` with empty stubs for every hook point:

```tcl
proc ipx_add_files      {}  { ... }   # add RTL sources to the edit project
proc ipx_merge_changes  {}  { ... }   # called after ipx::merge_project_changes
proc ipx_infer_bus_interfaces {} { ... }
proc ipx_add_params     {}  { ... }   # reorder / group HDL parameters in GUI
proc ipx_add_memory_map {}  { ... }
```

Edit the stubs before running `create-ip`. The hooks are sourced by Vivado at the appropriate stage, so they have full access to `[ipx::current_core]`.

### Creating an IP

```bash
xviv create-ip --ip ip_systolic
```

This runs Vivado in batch mode and performs, in order:

1. Scaffold the IP skeleton in `build/ip/ip_systolic_1_0/`
2. Strip the boilerplate AXI-Lite interface Vivado adds by default
3. Add your RTL sources
4. Infer AXI-Stream and AXI-MM bus interfaces
5. Expose HDL parameters in the customisation GUI
6. Wire AXI-Lite memory maps
7. Save `component.xml`

### Editing an existing IP

```bash
xviv edit-ip --ip ip_systolic
```

Opens the IP edit project in the Vivado GUI. Make changes, then close — the `component.xml` is saved automatically.

### Out-of-context synthesis

Add `--out-of-context` to synthesise the IP standalone, without top-level I/O buffers. This is required when packaging the IP for use in a block design.

```bash
xviv synthesis --top ip_systolic --out-of-context
```

OOC synthesis stops after routing and writes a stub Verilog file to `build/synth/ip_systolic/netlists/ip_systolic_stub.v`.

## Block designs

### Defining a BD

```toml
[[bd]]
name       = "system"
fpga       = "z2"
xdc        = ["srcs/xdc/system.xdc"]
hooks      = "scripts/bd/system_hooks.tcl"
export_tcl = "scripts/bd/system.tcl"
```

### Generating the hooks file

```bash
xviv bd-config --bd system
```

Creates `scripts/bd/system_hooks.tcl`. The key proc is `bd_design_config`:

```tcl
proc bd_design_config { parentCell } {
    global _bd_design_tcl

    if { [file exists $_bd_design_tcl] } {
        # Re-run exported TCL to recreate the BD non-interactively
        source $_bd_design_tcl
        xviv_refresh_bd_addresses
        validate_bd_design
        save_bd_design
        exit 0
    } else {
        # First run: open the GUI for interactive design
        start_gui
    }
}
```

On the first `create-bd` run the BD file does not exist yet, so Vivado opens the GUI. Build your design interactively, then export it.

### Creating a BD

```bash
xviv create-bd --bd system
```

### Exporting a BD

After finishing the design in the GUI, export it as a versioned re-runnable TCL script:

```bash
xviv export-bd --bd system
```

This writes a file like `scripts/bd/system_abc1234.tcl` (where `abc1234` is the current git SHA) and updates the `scripts/bd/system.tcl` symlink to point at it. Commit the versioned file; the symlink is the stable reference for future `create-bd` runs.

A `_dirty` suffix is appended to the tag when the working tree has uncommitted changes. Do not treat dirty exports as canonical.

### Editing a BD

```bash
xviv edit-bd --bd system
```

Opens the existing `.bd` file in the Vivado GUI.

### Generating output products

```bash
xviv generate-bd --bd system
```

Runs `generate_target all`, validates the design, generates the HDL wrapper, and copies it to `build/wrapper/`.

## Synthesis

### Defining a synthesis target

```toml
[[synthesis]]
top   = "system_wrapper"
fpga  = "z2"
xdc   = ["srcs/xdc/top.xdc"]
hooks = "scripts/synth/system_wrapper.tcl"
```

### Running synthesis

```bash
xviv synthesis --top system_wrapper
```

The full flow is: synthesise → place → route → write bitstream → write XSA. Outputs land in `build/synth/system_wrapper/`:

```
build/synth/system_wrapper/
├── post_synth.dcp
├── post_place.dcp
├── post_route.dcp
├── system_wrapper_abc1234.bit
├── system_wrapper_abc1234.xsa
├── system_wrapper.bit          ← symlink to latest
├── system_wrapper.xsa          ← symlink to latest
├── build.json
└── reports/
```

`build.json` is written only when the run completes successfully. It records the Vivado version, part, git SHA, elapsed time, and output filenames — useful for CI artefact tracking.

The git SHA is embedded in the bitstream via `BITSTREAM.CONFIG.USR_ACCESS`. Bit 31 is the dirty flag; bits 27:0 are the 7-digit hex SHA. You can read it back over JTAG at runtime.

### Report flags

```bash
xviv synthesis --top system_wrapper \
    --report-all          # enable all reports below
    --report-synth        # timing summary + utilisation after synthesis
    --report-place        # IO, clock utilisation, hierarchical util after place
    --report-route        # DRC, methodology, power, timing after route
    --generate-netlist    # functional and timing netlists for simulation
```

### Synthesis hooks

```bash
xviv synth-config --top system_wrapper
```

Creates `scripts/synth/system_wrapper.tcl` with five stubs called at fixed points in the flow:

```tcl
proc synth_pre      {}  {}   # before synth_design
proc synth_post     {}  {}   # after  synth_design, before place_design
proc place_post     {}  {}   # after  place_design, before route_design
proc route_post     {}  {}   # after  route_design, before write_bitstream
proc bitstream_post {}  {}   # after  write_bitstream
```

### Opening a checkpoint

```bash
xviv open-dcp --top system_wrapper --dcp post_route
```

Valid `--dcp` values: `post_synth`, `post_place`, `post_route`.

## Simulation

### Defining simulation sources

```toml
[sources]
rtl = ["srcs/rtl/**/*.sv"]
sim = ["srcs/sim/**/*.sv"]
```

### Compiling and running

```bash
# Compile only
xviv elaborate --top tb_systolic

# Compile and run for 10 µs
xviv elaborate --top tb_systolic --run 10us

# With a DPI shared library
xviv elaborate --top tb_systolic --so libdpi_model --dpi-lib ./build/libs
```

This invokes `xvlog` → `xelab` → `xsim` in sequence. Compiled artefacts go to `build/xviv/<top>/`.

### Waveform viewer

```bash
# Open the waveform database from the last simulation run
xviv open-wdb --top tb_systolic

# Open the simulation snapshot directly
xviv open-snapshot --top tb_systolic

# Reload waveform after re-running simulation (no need to close xsim)
xviv reload-wdb --top tb_systolic
```

`reload-wdb` and `reload-snapshot` communicate with a running xsim instance over a named FIFO at `build/xviv/<top>/control.fifo`. This means you can recompile and reload without closing the waveform window.

## MicroBlaze / Vitis

This section covers the embedded software flow for designs that include a MicroBlaze soft processor.

### Defining a platform and app

```toml
[[platform]]
name      = "mb_platform"
synth_top = "system_wrapper"    # derives XSA path from synthesis output
cpu       = "microblaze_0"
os        = "standalone"

[[app]]
name     = "firmware"
platform = "mb_platform"
template = "empty_application"
src_dir  = "srcs/sw/firmware"
```

`synth_top` tells xviv to look for the XSA at `build/synth/system_wrapper/system_wrapper.xsa`. Alternatively, specify an explicit path with `xsa = "path/to/file.xsa"`.

### Creating the BSP

```bash
xviv create-platform --platform mb_platform
```

Generates the Board Support Package using `hsi` (the Xilinx Hardware Software Interface). The BSP lands in `build/bsp/mb_platform/` and includes `libxil`, driver headers, and a Makefile.

```bash
xviv platform-build --platform mb_platform
```

Compiles the BSP with `make -j<ncpu>`.

### Creating and building an app

```bash
# Scaffold the app from a Vitis template
xviv create-app --app firmware

# Build it
xviv app-build --app firmware

# Build and print ELF section sizes
xviv app-build --app firmware --info
```

Your source files go in `srcs/sw/firmware/` (or whatever `src_dir` is set to). The build system compiles them against the BSP.

### Programming the FPGA

`hw_server` must be running. Start it from Vivado Hardware Manager, or standalone:

```bash
/opt/Xilinx/Vivado/2024.1/bin/hw_server
```

Then:

```bash
# Bitstream only
xviv program --bitstream build/synth/system_wrapper/system_wrapper.bit

# Bitstream derived from platform entry
xviv program --platform mb_platform

# Bitstream + ELF
xviv program --platform mb_platform --app firmware

# Explicit paths
xviv program \
    --bitstream build/synth/system_wrapper/system_wrapper.bit \
    --elf       build/app/firmware/Debug/firmware.elf
```

### Processor control

```bash
# Soft-reset the MicroBlaze (no reprogramming)
xviv processor --reset

# Print JTAG targets and register state
xviv processor --status
```

### JTAG UART

Read `xil_printf()` output over JTAG without a physical UART cable. Requires the MDM IP in the block design with `C_USE_UART = 1`.

```bash
xviv jtag-monitor --uart
```

Press `Ctrl-C` to stop.

## Common errors

**`FPGA part 'xc7z...' is not in the installed Vivado part catalog`** — the `part` string in `[fpga]` is wrong or that device family is not installed. Run `get_parts *z020*` in the Vivado TCL console to find the correct string.

**`BD file not found`** — `create-bd` has not been run yet, or the `build.bd_dir` path in `project.toml` does not match where the BD was created.

**`Hooks file is configured but does not exist`** — run `xviv ip-config --ip <name>` or `xviv bd-config --bd <name>` to generate the hooks file first.

**`No ELF found in build/app/<name>`** — run `app-build` before `program`.

**Synthesis completes but bitstream is wrong** — check `build.json` for the `sha_tag` field. If it ends in `_dirty`, the bitstream was built from uncommitted changes. Commit or stash before a production build.
